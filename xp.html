What is Extreme Programming (XP)?

Extreme Programming (XP) is an Agile software development framework that aims to produce higher-quality software and a higher quality of life for the development team. XP is the most specific of the Agile frameworks regarding appropriate engineering practices for software development.

It is called "Extreme" because it takes common-sense principles and practices to extreme levels. For example, if code reviews are good, XP does code reviews *continuously* (this is Pair Programming).

The Five XP Values

XP is built on a foundation of five values that guide the team's work and collaboration.

  • Simplicity: Do the simplest thing that could possibly work. This helps avoid waste and unnecessary complexity.
  • Communication: Emphasizes face-to-face communication, pair programming, and shared understanding over heavy documentation.
  • Feedback: Seek and deliver feedback rapidly. This comes from the code (via TDD), the customer, and the team.
  • Respect: Team members respect each other, the customer, and the code. This fosters a healthy, collaborative environment.
  • Courage: The courage to speak the truth, refactor code, and adapt to change.

Core XP Practices

Feedback Loop Practices

  • Pair Programming: All production code is written by two programmers sitting at one machine. This provides built-in code review and knowledge sharing.
  • Test-Driven Development (TDD): Write a failing automated test *before* you write the code to make it pass.
  • On-site Customer: A real, empowered customer sits with the team full-time to answer questions and set priorities.

Continuous Process Practices

  • Continuous Integration (CI): The team integrates their work frequently (multiple times per day). Each integration is verified by an automated build to detect errors quickly.
  • Small Releases: Release working software to customers in small, frequent increments.
  • Refactoring: Continuously improve the design of existing code without changing its functionality.

Shared Understanding Practices

  • Simple Design: The code should be the simplest, cleanest solution that meets the current requirements.
  • Collective Code Ownership: Any pair of programmers can improve any code at any time.
  • Coding Standards: The team agrees on and follows a common set of coding rules.

Team Welfare Practices

  • Sustainable Pace: The team works at a pace that they can sustain indefinitely, avoiding burnout and overtime.
